# ТЗ: трекер предметов для модулей убежища Escape from Tarkov

## 0. Что это и для кого

Веб-приложение для отслеживания, какие предметы нужно собрать для прокачки модулей убежища в Escape from Tarkov. Пользователей двое (я и друг), у каждого свой прогресс. Аккаунтов и общего сервера не нужно — прогресс хранится локально в браузере.

Главная ценность приложения: в любой момент видеть «что мне ещё осталось собрать по всему убежищу», с учётом того, какие уровни модулей уже построены и какие предметы уже найдены.

Стиль кода неважен — главное, чтобы всё работало корректно и было удобно пользоваться.

---

## 1. Технический стек

- **React + Vite + чистый JavaScript** (без TypeScript).
- **Material-UI (MUI)** с тёмной темой — для интерфейса.
- **Хранение состояния** — `localStorage` (без бэкенда).
- **Деплой** — статика на Render.
- **Данные** — статический `src/data/data.json`, сгенерированный из приложенной Excel-таблицы (см. раздел 3).

Бэкенд, авторизация, база данных — НЕ нужны.

---

## 2. Исходные данные (Excel)

К ТЗ приложена таблица `tarkov_hideout_requirements_2026-06-14.xlsx`. В ней три листа:

- **`Требования`** — основной лист, по строке на каждое требование. Колонки: `Модуль`, `Уровень`, `Тип`, `Требование`, `Кол-во`, `Найти в рейде`, `Примечание`, `Источник`.
- **`Свод предметов`** — уже посчитанная агрегация предметов. **Использовать ТОЛЬКО как эталон для самопроверки** парсинга и логики свода (см. раздел 8.1). В приложение этот лист не тащить — свод приложение считает само.
- **`Примечания`** — текстовые пояснения и дата. В приложение не нужен (можно вытащить дату для подписи «данные актуальны на …»).

Всего: 27 модулей, 450 строк требований.

## 3. Конвертация Excel → `data.json` (этап сборки)

Нужно один раз распарсить лист `Требования` в `src/data/data.json`. Сделать это python-скриптом (`scripts/build_data.py`, через `openpyxl`) или вручную — на усмотрение. Результат коммитится в репозиторий, приложение в рантайме Excel не читает.

### 3.1. Значение колонки `Тип`

Каждая строка относится к одному из типов. Их надо разделить на **собираемые предметы** и **прочие условия**:

| Тип | Куда идёт | Комментарий |
|---|---|---|
| `Предмет` | **Предмет** (обязательный) | Основные собираемые предметы. |
| `Опционально` | **Предмет** (флаг `optional: true`) | Тоже предмет с кол-вом и FIR, но НЕ обязательный (альтернативный материал). НЕ блокирует кнопку перехода на уровень. |
| `Валюта` | условие | Напр. «Dollars 25000». Не собирается. |
| `Торговец` | условие | Напр. «Skier LL3» — уровень лояльности торговца. |
| `Модуль` | условие | Напр. «Generator 3» — нужен другой модуль. |
| `Навык` | условие | Напр. «Endurance Level 1». |
| `Издание/условие` | условие | Напр. «Owning standard game edition». |
| `Ожидание` | условие | Напр. «Wait ~30 minutes after construction». |

То есть **собираемые предметы = строки с типом `Предмет` или `Опционально`**. Всё остальное — «прочие условия», показываются в карточке модуля, но без чекбоксов сбора.

### 3.2. Поле FIR (Found in Raid)

Колонка `Найти в рейде`:
- `Да` → `fir: true`
- `Нет` → `fir: false`
- `—` → строка не предмет (для условий поле FIR не используется).

### 3.3. Целевая структура `data.json`

```json
{
  "meta": {
    "date": "2026-06-14",
    "source": "https://escapefromtarkov.fandom.com/wiki/Hideout"
  },
  "modules": [
    {
      "id": "air-filtering-unit",
      "name": "Air Filtering Unit",
      "maxLevel": 1,
      "isEvent": false,
      "levels": [
        {
          "level": 1,
          "items": [
            { "name": "Gas mask air filter", "qty": 5, "fir": true,  "optional": false },
            { "name": "Military power filter", "qty": 5, "fir": false, "optional": false }
          ],
          "conditions": [
            { "type": "Валюта",   "text": "Dollars", "qty": 25000 },
            { "type": "Торговец", "text": "Skier LL3" },
            { "type": "Модуль",   "text": "Generator 3" }
          ]
        }
      ]
    }
  ]
}
```

Правила формирования:
- `id` — slug из названия модуля (нижний регистр, пробелы → дефис). Используется как стабильный ключ.
- `maxLevel` — максимальный уровень модуля из таблицы.
- `isEvent` — `true` только для модуля **Christmas Tree** (ивентовый/временный), у остальных `false`.
- `levels` — отсортированы по возрастанию; внутри `items` — обязательные сначала, опциональные после (или просто по порядку из таблицы, неважно).
- `conditions[].qty` указывать только для валюты; для остальных условий поле не нужно.

## 4. Модель состояния (КЛЮЧЕВОЙ раздел — прочитать внимательно)

Раздели данные на **статику** (`data.json`, не меняется) и **состояние пользователя** (хранится в localStorage). Состояние пользователя — всего два объекта:

### 4.1. `builtLevels` — какие уровни уже построены

```js
builtLevels = { "air-filtering-unit": 0, "bitcoin-farm": 2, ... }
```

Значение — номер уже построенного уровня. `0` = модуль не построен совсем. По умолчанию все по нулям.

### 4.2. `collected` — какие предметы уже найдены

**Самое важное во всём приложении.** Статус «найдено» — это свойство НЕ конкретной строки требования, а самого предмета, **общее для всех модулей**. Ключ предмета — пара «название + FIR»:

```js
function itemKey(name, fir) {
  return name + (fir ? "__FIR" : "__STD");
}

collected = { "Power supply unit__FIR": true, "Bolts__STD": true, ... }
```

Почему именно «название + FIR», а не просто название: один и тот же предмет встречается в таблице и как FIR, и как обычный (например, `Power supply unit` — FIR в Bitcoin Farm 1 и обычный в Bitcoin Farm 2). FIR-версию нельзя зачесть вместо обычной и наоборот — это разные «слоты». Поэтому это два разных собираемых предмета с разными ключами.

**Следствие (и в этом весь смысл такой модели):** оба окна читают и пишут статус «найдено» в один и тот же объект `collected` по `itemKey`. Поэтому синхронизация между окнами получается автоматически, никакой ручной «рассылки» отметок по модулям писать не нужно. Отметил в окне 2 → во всех модулях окна 1 уже отмечено, потому что это физически одно значение. И наоборот.

### 4.3. Что выводится из состояния (не хранить, а вычислять)

- **Список предметов к сбору** = предметы тех уровней модуля, что **строго выше** уже построенного (`level > builtLevels[moduleId]`). Уже построенные уровни в сбор не попадают.
- **Готовность уровня** к постройке = все *обязательные* (`optional !== true`) предметы целевого уровня отмечены в `collected`.

## 5. Окно 1 — «Модули»

Список всех 27 модулей (карточками или аккордеонами). Для каждого модуля:

### 5.1. Выбор построенного уровня
У названия модуля — селектор (выпадающий список или степпер) для выбора уже построенного уровня: от `0` («не построен») до `maxLevel`. Это напрямую меняет `builtLevels[moduleId]`. Доступно независимо от кнопки из п. 5.4.

### 5.2. Переключатель отображения уровней
Тумблер на модуле: **«по уровням»** / **«всё сразу»**.
- «по уровням» — предметы сгруппированы по уровням (показываются только уровни выше построенного).
- «всё сразу» — все оставшиеся (непостроенные) уровни объединены в один список.

### 5.3. Отображение предметов
Для каждого предмета показать: **название**, **количество** (`qty`), пометку **FIR** (заметный бейдж, если `fir: true`), и **чекбокс «найдено»** (читает/пишет `collected[itemKey]`). Опциональные предметы (`optional: true`) — с бейджем «опционально».

Также блоком «Прочие условия» показать `conditions` уровня (валюта, торговец, модуль, навык, издание, ожидание) — текстом, без чекбоксов.

### 5.4. Кнопка «Построить следующий уровень»
Когда все **обязательные** предметы целевого уровня (`builtLevels[moduleId] + 1`) отмечены в `collected` — показать кнопку «Построить уровень N» (или «Перейти на следующий уровень»). По нажатию: `builtLevels[moduleId] += 1`. После этого предметы построенного уровня уходят из сбора (фильтр из п. 4.3). Если модуль уже на `maxLevel` — кнопку не показывать.

### 5.5. Архивация внутри окна
Найденные предметы (отмеченные) уезжают вниз списка и визуально приглушаются (серые/полупрозрачные). Сортировка: сначала несобранные, потом собранные.

## 6. Окно 2 — «Свод по всему убежищу»

Единый список **всех предметов, которые ещё нужно собрать по всем модулям сразу**.

### 6.1. Как считается
Пройти по всем модулям и их уровням, взять только уровни `level > builtLevels[moduleId]`, собрать все предметы, **сгруппировать по `itemKey`** (название + FIR). Для каждой группы:
- `name`, `fir` — из ключа;
- `totalQty` — сумма `qty` по всем вхождениям;
- `usedIn` — список «модуль + уровень», где предмет нужен (для подсказки «где используется»);
- статус «найдено» — `collected[itemKey]`.

FIR-версия и обычная версия одного предмета — **две отдельные строки** в своде (разные ключи).

### 6.2. Отображение
По строке на предмет: **название**, **суммарное количество**, бейдж **FIR**, чекбокс «найдено», и (желательно) подсказка `usedIn`. Опциональные можно пометить бейджем.

### 6.3. Архивация и синхронизация
- Чекбокс пишет в тот же `collected[itemKey]`.
- Найденные уезжают вниз и приглушаются.
- За счёт общего ключа (раздел 4.2): отметка здесь автоматически отражается во всех модулях окна 1, и наоборот. Отдельной логики синхронизации писать не нужно.

## 7. Хранение, версионирование, экспорт

- Всё состояние (`builtLevels` + `collected`) сохранять в `localStorage` под версионированным ключом, например `tarkov-hideout-v1`. При загрузке — восстанавливать.
- Кнопка **«Сбросить прогресс»** (с подтверждением).
- Кнопки **«Экспорт»** и **«Импорт»** состояния в JSON (скачать файл / вставить) — чтобы делиться сетапом с другом и иметь бэкап.
- При несовпадении версии данных старый прогресс ломаться не должен: лишние/отсутствующие ключи просто игнорируются (никаких падений).

## 8. Визуальный стиль

Тёмная тема в духе официального сайта https://www.escapefromtarkov.com/?lang=ru — тёмный фон, приглушённый хаки/оливковый, янтарно-жёлтый акцент, угловатые рамки, моноширинные/строгие акценты в типографике. Ориентир по палитре (можно скорректировать):

- фон: очень тёмный, почти чёрный — `#0e0e0e` / `#1a1916`;
- поверхности (карточки): `#23211c` / `#2b2823`;
- акцент (кнопки, активные элементы): янтарный `#c7a26b` / `#9a8866`;
- FIR-бейдж — заметным контрастным цветом (например, тёплый красно-оранжевый), чтобы сразу бросался в глаза;
- текст: светлый тёплый серый.

Реализовать через кастомную MUI-тему (`createTheme` + `ThemeProvider`) в одном файле. Интерфейс адаптивный: должно быть удобно и на ПК, и на телефоне.

Два окна — через табы/роутер (например, «Модули» и «Свод»), переключение вверху.

### 8.1. Самопроверка свода
После реализации сравнить вычисленный приложением свод (при `builtLevels` все = 0 и пустом `collected`) с листом `Свод предметов` из Excel: суммы `totalQty`, разбивка FIR/обычные и список «где используется» должны совпадать. Это проверка корректности парсинга и агрегации.

## 9. Чеклист нюансов (грабли)

1. В сбор идут только типы `Предмет` и `Опционально`; всё прочее — условия без чекбоксов.
2. Опциональные предметы НЕ блокируют кнопку «построить уровень».
3. Ключ «найдено» = `название + FIR`, общий для всех модулей. Это даёт двустороннюю синхронизацию окон бесплатно.
4. FIR и обычная версия одного предмета — разные собираемые предметы (разные ключи, разные строки в своде).
5. Свод и список к сбору фильтруются по построенным уровням (`level > builtLevels`).
6. Бывают уровни без предметов (только условия) — список предметов должен корректно показывать пустоту.
7. Модуль на максимальном уровне — без кнопки перехода.
8. Christmas Tree (`isEvent: true`) — добавить тумблер «показывать ивентовые модули» (по умолчанию можно показывать).
9. localStorage версионировать; обновление данных не должно ломать старый прогресс.
10. localStorage не работает в предпросмотре артефактов Claude.ai — проверять только в реально запущенном/развёрнутом Vite-приложении.

## 10. Критерии готовности (Definition of Done)

- [ ] Окно «Свод»: агрегация по всему убежищу с учётом построенных уровней, разделение FIR/обычные, архивация.
- [ ] Отметка «найдено» в одном окне мгновенно отражается в другом (через общий `collected`).
- [ ] Прогресс сохраняется в localStorage и переживает перезагрузку.
- [ ] Есть сброс и экспорт/импорт прогресса.
- [ ] Тёмная тема в стиле Tarkov, адаптивно под телефон и ПК.
- [ ] Собирается и деплоится как статика на Render.
